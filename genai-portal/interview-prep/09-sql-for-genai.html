SQL & Data Systems for GenAI Roles

Prepare for the data work behind production GenAI: reliable joins, analytical SQL, query tuning, transactional workflows, vector search and safely generated queries.

12 high-signal questionsBackend + GenAI relevanceWrite, explain, optimize
A strong GenAI data design combines relational correctness, vector similarity, metadata filters and authorization before context reaches the model.
⌁
What interviewers are checking

Do not stop at valid syntax. State the table grain, expected cardinality, correctness risks, indexes, authorization boundary and how you would inspect the query plan.

12 questions
01How do you reason about joins without accidentally duplicating rows?joinscardinalitygrain＋
Strong answer

I state the grain of every table before writing the join: one row per customer, document, chunk or event. Then I identify whether the relationship is one-to-one, one-to-many or many-to-many and predict the output grain. If I only need existence, I prefer EXISTS; if I need an aggregate from the many side, I aggregate before joining. I validate with row counts and uniqueness checks rather than using DISTINCT to hide a modeling mistake.

30-second version

Define the input and output grain, know the join cardinality, and aggregate the many side before joining when necessary. DISTINCT is not a repair strategy.

Likely follow-up: When is a many-to-many join intentional?
02When would you use a window function instead of GROUP BY?windowgroup-byanalytics＋
Strong answer

GROUP BY collapses rows to one row per group. A window function preserves each row while calculating a value over related rows, which is useful for ranking, running totals, moving averages, percentiles and comparing an event with a group average. I also explain the partition, ordering and frame because the default frame can change results for cumulative calculations.

rank_queries.sql
SELECT tenant_id,
       query_text,
       latency_ms,
       ROW_NUMBER() OVER (
         PARTITION BY tenant_id
         ORDER BY latency_ms DESC
       ) AS latency_rank
FROM query_traces;
30-second version

Use GROUP BY to reduce rows and windows to keep row detail while adding group-aware calculations.

Likely follow-up: What is the difference between ROW_NUMBER, RANK and DENSE_RANK?
03How do you return the latest record per document or user?deduprow-numberlatest＋
Strong answer

I use ROW_NUMBER() partitioned by the entity key and ordered by the business timestamp plus a deterministic tie-breaker such as an immutable ID. Then I filter to row number one. PostgreSQL also supports DISTINCT ON, but I use it only when the team is comfortable with PostgreSQL-specific syntax and the ordering rule is explicit.

latest_document.sql
WITH ranked AS (
  SELECT d.*,
         ROW_NUMBER() OVER (
           PARTITION BY document_id
           ORDER BY indexed_at DESC, version_id DESC
         ) AS rn
  FROM document_versions AS d
)
SELECT *
FROM ranked
WHERE rn = 1;
30-second version

Rank rows within each entity by timestamp and a stable tie-breaker, then keep rank one.

Likely follow-up: Why is ordering only by timestamp sometimes nondeterministic?
04When do you use a CTE, subquery or temporary table?ctesubquerytemp-table＋
Strong answer

I choose based on clarity, reuse and execution behavior. A subquery is fine for a small local transformation. A CTE gives a named reasoning step and supports recursion; in modern PostgreSQL it may be inlined unless materialization is requested. A temporary table is useful when an expensive intermediate result is reused, needs indexes, or must be inspected across several statements. I confirm performance with the actual plan instead of assuming one form is always faster.

30-second version

Use the simplest form that keeps the query understandable; move to a temporary table when an intermediate result is reused or needs its own indexes.

Likely follow-up: What problem does a recursive CTE solve?
05How do you find the top N rows per tenant, category or model?top-nwindowranking＋
Strong answer

I rank rows within each group using a window function and then filter to the required rank. The ordering must match the business rule, and ties need an explicit decision. For large datasets I consider an index starting with the partition key and ordering columns, but I still inspect whether the planner can use it efficiently.

top_failures.sql
WITH ranked AS (
  SELECT tenant_id, failure_reason, failure_count,
         DENSE_RANK() OVER (
           PARTITION BY tenant_id
           ORDER BY failure_count DESC
         ) AS rnk
  FROM daily_agent_failures
)
SELECT *
FROM ranked
WHERE rnk <= 3;
30-second version

Rank inside each group, decide how ties should behave, and filter to N.

Likely follow-up: How would you return exactly three rows even when values tie?
06How do indexes help, and how do you use EXPLAIN ANALYZE?indexesexplainperformance＋
Strong answer

An index trades storage and write cost for faster lookup, join or ordering paths. I choose indexes from real predicates and access patterns, considering column order, selectivity, covering columns and partial indexes. With EXPLAIN ANALYZE I compare estimated versus actual rows, identify sequential scans, expensive sorts, nested loops over large inputs, repeated work and buffer usage. I test with production-like data because a small development table can produce a misleading plan.

30-second version

Index the filters and joins the application actually uses, then verify with actual rows, timing and buffers—not intuition.

Likely follow-up: Why might PostgreSQL correctly choose a sequential scan even when an index exists?
07What transaction and isolation concepts matter in a GenAI workflow?transactionsisolationidempotency＋
Strong answer

A transaction makes a set of database changes atomic, but I avoid holding it open across a slow model or tool call. I persist an operation or outbox record, commit, call the external service, and then update state idempotently. Isolation level controls which concurrent changes are visible; I use locking or optimistic version checks only around the small critical section that requires consistency. Every retryable workflow needs a stable idempotency key and a uniqueness constraint.

30-second version

Keep transactions short, never wrap a long LLM call inside one, and make retried state transitions idempotent.

Likely follow-up: What anomaly can occur with read-then-write logic under concurrency?
08When would you use JSONB versus normalized relational tables?jsonbnormalizationschema＋
Strong answer

I normalize stable entities, relationships and fields that need constraints, joins or frequent filtering. JSONB is useful for sparse provider metadata, evolving tool payloads or trace attributes, but important fields should not disappear into unvalidated blobs. PostgreSQL can index JSONB paths, yet updates may rewrite larger values and application-level schema drift can become difficult to govern. A common design keeps core columns relational and stores optional extension data in JSONB.

30-second version

Normalize fields with business meaning and constraints; use JSONB for genuinely flexible attributes, not as a substitute for data modeling.

Likely follow-up: How would you validate the shape of JSONB data?
09How would you combine relational filters and vector search with pgvector?pgvectorfiltersretrieval＋
Strong answer

I store each chunk with its embedding plus relational metadata such as tenant, document, source, access policy and freshness. The query first enforces authorization and selective metadata filters, then orders candidates by the chosen vector distance. For scale I compare exact search with HNSW or IVFFlat, measure recall and latency, and consider hybrid retrieval by combining lexical and vector scores. I never treat vector similarity as an authorization mechanism.

authorized_vector_search.sql
SELECT chunk_id, document_id, content,
       embedding <=> :query_embedding AS distance
FROM document_chunks
WHERE tenant_id = :tenant_id
  AND access_level <= :user_access_level
  AND is_active = TRUE
ORDER BY embedding <=> :query_embedding
LIMIT 20;
30-second version

Keep embeddings beside enforceable metadata, apply tenant and policy filters, then rank authorized candidates by vector distance.

Likely follow-up: What trade-off does an approximate vector index introduce?
10How do you design a safe text-to-SQL system?text-to-sqlsecurityvalidation＋
Strong answer

I give the model a curated semantic schema rather than unrestricted database access. Generated SQL is parsed into an AST and checked against an allowlist of statements, schemas, tables, columns and functions. The database role is read-only and scoped to authorized rows, with statement timeout, row limit and resource controls. I reject multiple statements and writes, log the prompt and approved query, and require confirmation for sensitive reports. Parameterization protects user-supplied values, but generated identifiers still require structural validation.

30-second version

Constrain the schema, validate the SQL structure, execute with a least-privilege read-only role, cap cost and rows, and preserve an audit trail.

Likely follow-up: Why is a prompt saying “only write SELECT” insufficient?
11How do you enforce tenant isolation in the database?multi-tenantrlsauthorization＋
Strong answer

Every tenant-owned row carries a tenant key, and repository methods require tenant context rather than accepting it optionally. Composite unique constraints include the tenant key, and indexes start with it for common access paths. PostgreSQL row-level security can provide defense in depth, but the application must set trusted session context correctly and use a role that cannot bypass policy. I test cross-tenant access explicitly, including generated SQL and vector retrieval paths.

30-second version

Make tenant context mandatory in schema, queries, constraints and tests; use RLS as an additional database boundary, not the only one.

Likely follow-up: What is dangerous about connection pooling with session-level tenant settings?
12How do you evaluate LLM-generated SQL?evaluationcorrectnessperformance＋
Strong answer

I build a versioned set of natural-language questions with schema context, expected result semantics and edge cases. Exact SQL string match is too strict, so I compare execution results on controlled fixtures, inspect whether required filters and joins are present, and score safety and cost separately. Tests include ambiguity, empty results, duplicate-producing joins, time zones, nulls, authorization boundaries and adversarial requests. Before production execution I run static validation and optionally a cost-limited plan check.

30-second version

Evaluate result correctness, authorization, safety and query cost on realistic fixtures—not only whether the generated SQL resembles a reference string.

Likely follow-up: How would you handle two different queries that return the same correct result?
